Ruby Classes vs. JavaScript Constructors

2/11/16

First, let's talk about what Ruby and JavaScript have in common. They are both object-oriented languages, meaning they are based on objects; data structures that each contain their own ecosystem of attributes and methods. In order to complete a task in an object oriented language, you must create and manipulate these objects.

Go in to further detail and Ruby and JavaScript differ. Ruby utilizes class-based programming (aka class-orientation), while JavaScript utilizes prototype-based programming (aka prototype-orientation or instance-based programming). What is class-orientation? It is a style of object-oriented programming, that allows you to create classes of objects. All objects must be some instance of a class. When you create a new string, for example, that string is simply a child of the parent String class. That string has a set of pre-defined methods that cannot be changed. This is the most popular type of object-oriented programming (OOP). Just like the puppies below are children of the golden retriever, Ruby objects are children of their parent class.

Ruby Classes - Objects are Instances of Classes

While class-based programming allows you to create classes of objects, prototype-based language allows you to create clones of existing objects. The same type of parent / child relationship does not exist in prototype-based language. When you create a new object, a clone of a previous object, you have full control over that object's attributes and methods. While the newly cloned object will inherit the same properties and methods from the original object, you can actually create or remove the new object's methods. While Ruby objects are instances of their parent class, just as the dogs in the photo above are children of their mom, JavaScript objects are clones of another object declared through a constuctor. The clones are similar, they initially inherited the same attributes and methods, however, they are more flexible in adding methods or attributes after they have been initially defined.

JavaScript Constructors - Objects are Clones of Objects

For example, let's consider a farm full of animals, where each animal is a class / constructor. In Ruby, cows, chickens, and pigs would each be a child of the animal class. The types of attributes and methods they could have would be inherited from this class. In JavaScript, cows, chickens, and pigs would be clones of the animal constructor, with each animal extended to have attributes and methods specific to themselves in addition to the ones inherited when cloned.

This increased control with prototype-orientation comes at a cost - decreased organization. With prototype-orientation, there is more potential for customzation. You can change the methods and attributes of each individual object after it has been created. The heirarchy of parent vs childen is less clear. When objects are simply clones of other objects, who is the parent? The child? Where is the order? With class-orientation, that relationship is clear. All objects are children of some parent class. However, while class-orientation provides us with a clear heirarchy and therefore more strucutred code, prototype-orientation hands developers the freedom to build more powerful programs. In a progrmaming type defined as object-oriented, more control over your objects means more control over your code.

This brings us back to the difference between Ruby classes and JavaScript constructors. Now that we have an increased understanding of class vs. prototype orientation, we can understand that Ruby classes are built to create objects that are instances of a class, while JavaScript constructors are built to be cloned. While they are similar in many ways, both contain basic attributes and methods that will be inherited by their children, they differ in their purpose and flexibility. It all boils down to this: Ruby classes allow you to create objects that are instances of that class, that inherit all their attributes and methods. JavaScript constructors allow you to create clones of an object, that inherit all that object's attributes and methods, while giving you the freedom to add/remove attributes and methdods after the object is created.

It's as simple as that! There are, of course, many differences in syntax. Those are just semantics. If you truly understand why Ruby classes and JavaScript constructors are different, you will have a much better grasp of the two languages as a whole.
